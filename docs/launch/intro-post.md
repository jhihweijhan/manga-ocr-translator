# 介紹貼文草稿

## 建議標題

我做了一個本機漫畫 OCR 翻譯工具，重點不是全自動，而是讓你看得出模型讀錯了什麼

## Blog / HN copy

我一直覺得 OCR + 翻譯工具最危險的地方，不是翻得很爛。

翻得很爛時，你通常看得出來。真正麻煩的是另一種情況：OCR 把原文讀錯了，翻譯模型卻產出一段看起來很順的譯文。結果不是「模型失敗」，而是使用者很難知道自己正在相信一個錯誤的來源。

Manga OCR Translator 是我為這個痛點做的小工具。它不是要把漫畫翻譯變成全自動流程，也不是 scanlation 製作工具。第一版只處理單張圖片，目標是讓本機 OCR 和翻譯結果變得可檢查、可校對。

流程很直接：

1. 在瀏覽器上傳一張 PNG、JPEG 或 WebP 圖片。
2. 從本機 Ollama 模型清單選 OCR 模型與翻譯模型。
3. 工具自動跑 OCR，接著跑翻譯。
4. 你可以逐段檢查 OCR 原文與區塊譯文。
5. 如果 OCR 原文有錯，可以修正原文，再只重新翻譯。
6. 需要留存或接續校對時，可以匯出 task JSON，之後再匯入還原文字區塊、區塊譯文、模型、語言與提示詞設定；也可以複製譯文或匯出純譯文 TXT。

我刻意把 OCR 原文和區塊譯文分開顯示，因為「可檢查」比「看起來完成」更重要。尤其是漫畫圖片裡的短句、人名、語氣詞或排版特殊文字，OCR 一旦讀錯，後面的翻譯很容易跟著歪掉。

這也是為什麼工具走 local-first。模型請求透過本機 backend 轉送到本機 Ollama；backend 不保存上傳圖片，也沒有任務歷史。這能降低資料外流與儲存風險，但我不會把它包裝成完全私密或完全安全：圖片仍會經過 browser、本機 backend 與 Ollama。比較準確的說法是，它把模型執行與資料路徑留在你的本機環境，並避免 backend 持久保存上傳圖片。

目前它不支援多張圖片批次或多頁閱讀任務、不處理 CBZ/ZIP、不做串流進度、不把譯文嵌回圖片、不做 on-image overlay，也不保存任務歷史。task JSON 匯入/匯出只保存文字區塊、區塊譯文、模型、語言與提示詞設定，不包含原圖。這些限制是刻意留白，因為我想先把最小但重要的流程做好：單張圖片、可檢查的文字區塊、可校對的 OCR 原文、校對後重新翻譯，以及可攜出的文字結果。

我最想聽到的回饋不是「能不能一次處理整本」。那當然有用，但不是第一個問題。第一個問題是：當你用本機模型處理漫畫 OCR 或翻譯時，哪個步驟最需要人工檢查？是 OCR 文字、人名、語氣、擬聲詞，還是最後的區塊譯文？

如果這個工具能幫你更快發現模型讀錯了什麼，而不是讓錯誤藏在流暢譯文裡，那它就有價值。
